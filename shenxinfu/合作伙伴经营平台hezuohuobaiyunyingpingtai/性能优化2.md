落地方案说明
一、秒级单点登录 + 数据中心预渲染 + SCP 构建分包策略优化

目的：解决用户跳转数据中心页面加载过长问题，而跨系统访问标准是端到端渲染时间需要达到 <4s；

现状分析：用户跳转数据中心加载耗时包括两部分，单点登录（3次接口重定向）+ SCP页面渲染；而目前单点登录需要等待 `/regions` 接口调用成功后执行（实测大概需要7s左右），若用户登录后立刻通过导航快速跳转，预加载就失效了；

优化方向：用户登录成功后立刻执行单点登录对数据中心的资源进行预渲染；

秒级单点登录

目前线上用户接入基本都是1-3个数据中心，利用这一点，可以将用户的前3个数据中心的数据通过cgi提前注入html，用户登录后立刻循环执行预加载（单点登录），这里加载不同域的资源不会阻塞其它资源加载；实现用户登录后秒级单点登录；

预渲染方案

方案探索

方案一：在SCC插入一个隐藏的iframe在屏幕外加载数据中心，利用浏览器缓存去预渲染；

缺点：iframe会造成当前窗口资源抢占，SCC卡顿无法操作；



方案二：利用谷歌浏览器 Speculation rules API 访问数据中心预渲染；

需要使用https才生效，涉及不同源问题（数据中心源不确定），需要后端配合添加响应头 `Supports-Loading-Mode: credentialed-prerender`

缺点：scp与scc单点登录相关api都需要添加响应头，涉及安全合规问题；若浏览器不支持，预加载这个优化效果也没有了，该方案不是最优的；



方案三：预加载+预渲染（方案二）；

增加预渲染数据中心策略，预加载改成使用fetch执行，在预加载执行成功后再利用谷歌浏览器 Speculation rules API 访问数据中心预渲染；

缺点：依赖浏览器版本，有兼容性问题；谷歌浏览器110+，edge110+，不支持safari和firefox

但结合线上 Sentry上报用户使用浏览器情况统计（2024 年 1 - 4 月），能够达到最佳预渲染效果的用户占比达 71%

动态CDN

方案一：Vite 实验性功能 renderBuiltUrl

支持将返回的 JS 或者图片文件等路径包一层回调返回带 cdn 的路径。但存在一个问题，CSS 里面的图片文件无法通过 renderBuiltUrl 回调修改路径，可行的方案是需要 CSS in JS。

解决思路是封装成一个 vite-plugin-inject-css-to-js 插件将 css 代码转换后注入到各个 js 文件中，这时候其实已经可以在该插件中新增一个 beforeInjectCss 的回调方法，将 cssCode 里面的图片文件路径进行替换。

缺点：实验性功能不太适合用于生产环境



方案二：

考虑到只是做一层路径转换，利用 Vite 丰富的插件能力，最后编写一个插件去处理，对静态资源做路径转换；

子应用动态加载

原来首屏即加载所有子应用资源文件，包括fees\bss\csm三个子应用共20+个静态资源；通过动态注册子应用路由方案，在用户点击对应路由后对子应用资源进行动态加载，这样就可以减少首屏资源；具体落地详见文档 SCC子应用动态加载

合并@share/services服务

原@share/services下有10+个服务，每个服务单独打包为1个js文件，都是无状态的；通过构建优化，将原来10+个js文件合并为1个js文件；具体落地详见文档：scc @shared/services 服务优化

图片资源压缩

开启前端gzip，在打包时就压缩，缩短请求资源时服务端进行动态压缩的响应时间；

图片资源控制，利用webp格式图片的优点，使用WebP 转换工具将所有png/jpg/jpeg转换保存一份webp格式图片，结合 Nginx 实现图片资源动态加载，消除大图；
