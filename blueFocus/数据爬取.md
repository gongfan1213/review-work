# 小红书数据爬取方案

## 1. 可爬取的数据类型

小红书上的笔记（帖文）通常包含文本、标签、图片和视频等内容。例如，通过官方接口“feed”可获取笔记详情，其中包含笔记文本、标签、点赞收藏数、评论数、作者信息、图片/视频链接等。一个笔记记录的核心字段一般包括：标题、笔记ID、链接、作者昵称、作者ID、发布时 间等。此外，可以访问用户主页获取用户资料（昵称、粉丝数、关注数等）以及该用户发布的笔记列表、点赞笔记、收藏笔记等。对于评论，接口提供获取笔记一级评论和二级评论的功能，可爬取评论内容、评论者昵称、评论时间等信息。综上，可爬取的数据类型包括：笔记内容（文字、标签）、笔记配图或视频链接、作者/用户信息、评论内容及互动统计等。下图展示了小红书某话题下的笔记示例，其中包含文字和视频内容。

## 2. Python 库、框架与开源项目

可使用**Python**主流爬虫库和框架，如`requests`（发送HTTP请求）、`BeautifulSoup`/`lxml`（解析HTML）、`Scrapy`（高效异步爬虫框架）、`Selenium`或`Playwright`/`Pyppeteer`（浏览器自动化模拟）等。对于签名和加密，可以使用`execjs`在Python中执行小红书官方的JS签名函数。如果要分析网页元素或抓取动态内容，也可使用基于Chromium的库如`DrissionPage`。常见的开源项目包括：[cv-cat/Spider\_XHS](https://github.com/cv-cat/Spider_XHS)（专门面向小红书的爬虫解决方案）、charstal/xhs\_simple\_crawler（简单示例）、NanmiCoder/MediaCrawler（多平台数据抓取，采用Playwright等自动化工具）等。这些项目通常封装了登录、API调用及签名处理逻辑，可参考其实现并二次开发。

* **Scrapy框架**：适用于高并发抓取。可定义Spider并使用管道保存数据，例如有教程演示使用Scrapy递归抓取小红书图文频道并保存图片。Scrapy内置异步处理，可结合中间件使用代理池、UA池；
* **Requests/Selenium**：Requests适合简单接口调用，对于需要模拟完整浏览器或动态登录的场景可用Selenium（配合无头Chrome）或`Playwright`；一些示例用Selenium结合抓包工具爬取小红书数据。
* **执行签名**：小红书请求需要签名参数（如`x-s`、`x-t`）和`xsec_token`，可通过逆向得到的JS（如`getXs`函数）配合`execjs`生成；也可参考第三方签名工具库。
* **开源项目**：除了上述Spider\_XHS和MediaCrawler外，还有Big-Buffer/XiaohongshuSpider（演示通过Appium+mitmproxy获取图片的思路）等，均可作为参考。

## 3. 模拟移动端请求的技术路径

小红书移动端接口多通过加密请求访问，可通过抓包和接口分析来获取参数。常用抓包工具包括：**Fiddler、Wireshark、Charles、mitmproxy**等。具体流程举例：在手机端打开小红书，通过分享功能复制笔记链接，然后在电脑浏览器中打开该链接并启用开发者工具，可在Network标签中找到后台返回的接口请求和响应。也可使用安卓模拟器（如夜神/Nox）配合Fiddler或mitmproxy，将手机代理到抓包工具，直接截获App发出的HTTP/HTTPS请求。抓到接口后，可以在Python中模拟对应请求，注意按接口要求添加签名、时间戳、设备参数等。必要时可使用Appium等自动化工具驱动App执行操作，再结合mitmproxy获取请求。总之，模拟移动端请求的关键在于：**获取App的网络请求接口、逆向签名算法、设置正确的请求头和参数**。

## 4. 登录或 Token 验证机制

部分数据接口需要登录态或特定token才返回。常见做法是**先登录小红书账号**以获得Cookie，再带Cookie请求接口。例如，Spider\_XHS项目说明在 `.env` 文件中填入浏览器登录后获得的 Cookie（需包含字段`a1`和完整Cookie字符串）。可以用Web端登录（如小红书网页版或PC端）后在开发者工具中复制Cookie，或者使用手机扫码、短信验证码登录官方App/PC客户端获取Cookie（如Spider\_XHS支持扫码登录和短信登录）。对于API请求，还需计算签名参数：可将App端的签名JS提取出来，用`execjs`在Python中调用生成`x-s`、`x-t`等。获取`xsec_token`一般由前序接口返回。此外，要注意**Cookies轮换**：同一Cookie访问次数过多会触发频率限制，需要准备多份Cookie并定期切换。

## 5. 小红书常见反爬机制及规避方法

小红书对爬虫有多重防护：

* **签名验证**：接口参数（`x-s`、`x-t`）需正确加密签名，否则返回`success:false`。因此必须逆向小红书的签名逻辑并在请求中携带正确签名。
* **频率限制**：同一IP或Cookie短时间大量请求会被封禁，并可能出现“疑似异常流量”提示。实测128并发、10万次调用后账号即被限流。规避方法包括：减慢请求频率（`time.sleep`）、并发限制、分布式爬取；对接口进行合理分页和限量请求；监测返回状态（如`success==false`、错误码461）及时切换Cookie。
* **验证码/人机验证**：高频或异常访问可能触发滑块验证码或人机检测。这时常用策略是降低频率、多使用不同IP、改换访问路径或休眠后重试。
* **设备指纹检查**：小红书会检查请求中的设备/浏览器指纹参数（如Spider\_XHS中的`b1`字段）。可尝试固定指纹值、或模拟真实设备信息，以减少异常特征。
* **HTTPS证书固定/防翻墙技术**：小红书App可能存在防抓包措施（如校验HTTPS证书、检测代理）。可使用mitmproxy证书安装到模拟器上，或采用更可靠的抓包方式（官方模拟器代理、真机USB调试配合adbproxy等）。

综合以上，常见**规避方法**包括：使用合理的请求间隔并动态调整频率、轮换多个Cookie和账号；使用代理IP（尤其是中国大陆IP）和不同User-Agent、浏览器指纹等模拟真实用户；对加密参数采用正确签名；以及在遇到验证码时尽量避免进一步请求，改换IP/账号或手动处理。

## 6. 代理池、UA 池与请求调度

为了提高爬取成功率，应使用**代理池**和**UA池**来分散风险。常用做法是准备多个HTTP代理（可使用付费的高匿住宅IP或ISP代理）来替换请求源IP；请求时随机切换User-Agent字符串模拟不同设备。许多爬虫框架（如Scrapy）支持中间件自动分配代理和UA。根据具体需求选择异步并发方案：Scrapy内置Twisted异步，可极大提升并发量；也可使用Python的`aiohttp`等库自行实现协程并发。对于批量抓取，建议使用任务调度器（如Scrapy的调度）或者多线程/多进程（注意Python全局锁）来控制并发。爬取小红书时尤其需要低速率并发，以避免触发平台检测。例如，在Scrapy中可以在每个请求中设置`meta`字段使用代理。总之，结合**代理池+UA轮换+异步调度**是常用提升爬取稳定性的策略。

## 7. 法律与合规建议

任何爬虫行为都需遵守法律法规和平台规则。《最高检》报道指出，爬虫使用不当可能侵犯个人信息权利，如“非法获取公民个人信息”可构成犯罪；使用爬取数据实施违法行为（如侵犯著作权、商业秘密等）同样受刑法规制。因此应关注以下要点：首先，**合法性**：仅抓取公开可获取的数据，避免收集未经允许的个人敏感信息；遵守《网络安全法》《个人信息保护法》等规定，确保爬虫手段不突破访问控制或解密平台加密。其次，**遵守robots协议**：网站通常通过Robots协议指明允许抓取的范围，超过约定范围可能带来法律风险。虽然小红书未公开API，但应尽量尊重其技术限制和使用条款，不进行“镜像服务”式的大规模数据搬运。再次，**尊重隐私与透明性**：若采集的内容涉及用户隐私，应获得授权或脱敏处理；公开使用前明确说明用途、遵循公平使用原则。最后，**合规运营**：应提醒团队或客户尊重小红书的用户协议和社区规定，不进行超过平台使用规定的数据操作。总之，爬虫应当以合法合规为前提，重视数据安全与隐私保护，避免对被爬取平台造成损害。
