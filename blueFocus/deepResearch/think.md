好的，下面将**详细分析本项目的“思考”流程**，即：当用户输入一个问题后，系统是如何“思考”、推理、决策并逐步生成高质量答案的。这里的“思考”不仅仅是 LLM 的单步输出，而是**多步推理、反思、检索、评估、聚合等一整套智能流程**。

---

# 一、整体思考流程概览

1. **接收用户问题**  
   用户通过前端/API输入问题，后端接收并标准化处理。

2. **初始化推理上下文**  
   包括 token 预算、历史知识、URL、允许的 action、评估指标等。

3. **多步推理主循环**  
   系统会多轮循环，每轮都“思考”下一步最优行动（answer/search/reflect/visit/coding）。

4. **动态决策与反思**  
   每轮决策后，系统会评估结果，必要时反思、细化问题或切换策略。

5. **答案聚合与后处理**  
   多步推理后，系统聚合所有知识，生成最终答案并格式化输出。

---

# 二、关键“思考”节点详细分析

## 1. 动态决策主循环（getResponse）

- **核心函数**：`getResponse`（见 `src/agent.ts`）
- **流程**：
  1. 设定主问题为 gaps，初始化知识库。
  2. 进入循环，只要 token 未超预算就持续推理。
  3. 每轮动态生成 prompt，明确本轮允许的 action。
  4. 调用大模型（LLM）决策下一步 action 及参数。
  5. 根据 action 类型进入不同分支（见下文）。
  6. 每轮后评估结果，必要时反思、细化、调整策略。
  7. 若多轮失败，自动进入 beast mode，强制输出答案。

---

## 2. “思考”类型的 action 及其流程

### a. answer（直接回答）

- LLM 认为已有足够知识，尝试直接作答。
- 系统自动评估答案质量（evaluateAnswer），如不合格则生成改进建议，驱动反思。

### b. search（外部检索）

- LLM 认为需要更多信息，自动生成搜索关键词，调用外部搜索引擎检索。
- 新知识补充到知识库，供后续推理使用。

### c. reflect（反思/子问题生成）

- LLM 认为当前知识有缺口，自动生成与主问题相关的子问题（gaps）。
- 系统将子问题补充到 gaps，驱动多步推理。

### d. visit（网页内容抓取）

- LLM 选择访问高相关 URL，抓取网页内容，丰富知识库。

### e. coding（代码求解）

- LLM 认为问题为代码相关，自动调用代码沙盒求解。

---

## 3. 反思与自适应机制

- 每个问题分配多种评估类型（如完整性、时效性、权威性等）。
- 评估失败自动生成改进计划，驱动下一步反思或细化。
- 若连续多轮评估失败，自动进入 beast mode，放宽约束，优先输出任何可用答案，保证鲁棒性。

---

## 4. “思考”内容的流式输出

- 在 `src/app.ts` 中，系统会将 LLM 的“思考”内容（如推理过程、反思、分步答案等）分块流式推送给前端。
- 前端实时渲染，用户可以看到 AI 的“思考过程”，而不是只看到最终答案。

---

# 三、代码中的“思考”体现

- **Prompt 设计**：每轮 prompt 都会引导 LLM 先“思考”再行动（如 `<action-reflect>`、`think` 字段）。
- **StepAction/AnswerAction**：每一步的结果都包含 `think` 字段，记录 LLM 的推理、理由、反思等。
- **流式输出**：每个 chunk 的 `delta` 字段中，`type: "think"` 表示这是“思考”内容，前端可区分渲染。

---

# 四、举例说明

假设用户问：“如何高效学习机器学习？”

1. **第一步**：系统先“思考”是否能直接回答，若不能，自动生成子问题（如“机器学习的核心知识点有哪些？”、“有哪些高效学习方法？”）。
2. **第二步**：针对每个子问题，系统“思考”是否需要检索外部信息，或直接作答。
3. **第三步**：每步推理后，系统评估答案质量，必要时反思、细化问题。
4. **最终**：系统聚合所有“思考”与答案，输出给用户，并以流式方式逐步展现推理过程。

---

# 五、总结

本项目的“思考”流程是**多步推理+动态决策+反思自适应+流式输出**的有机结合。  
每一步都不是简单的 LLM 输出，而是结合了外部检索、自动评估、反思、子问题生成等多种智能机制，最终让用户看到的不只是答案，更是 AI 的“思考过程”。
