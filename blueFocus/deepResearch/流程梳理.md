# 用户输入问题后的分析流程

## 1. 用户输入问题

- 用户通过 Web 前端、API 或命令行等方式输入一个自然语言问题。
- 该问题会被传递到后端服务（如 `src/app.ts`），最终调用 `getResponse` 这个核心函数。

---

## 2. 预处理与上下文准备

- **去除多余空格**，标准化输入。
- **历史消息处理**：如果有多轮对话，提取最新一轮的用户问题。
- **Schema 初始化**：根据问题内容和语言，动态生成用于约束 LLM 输出结构的 schema（如允许哪些 action）。
- **上下文对象初始化**：包括 token 预算追踪（防止超限）、action 追踪（记录每步决策）、知识库、URL 缓存等。

---

## 3. 问题与知识初始化

- **gaps 初始化**：将主问题加入待解决问题列表（gaps），后续可动态扩展为子问题。
- **allQuestions/allKnowledge**：分别记录所有遇到的问题和已获得的知识。
- **URL 提取**：自动从用户输入和历史消息中提取 URL，便于后续外部内容抓取。

---

## 4. 多步推理主循环

### 4.1 动态决策循环

- 只要 token 未超预算，循环执行以下步骤：

#### a. 动态生成 Prompt
- 根据当前 gaps、知识、URL、允许的 action（如 answer/search/reflect/visit/coding）等，动态生成系统 prompt，指导 LLM 决策。

#### b. 组织消息上下文
- 将历史知识、用户消息、改进建议等合并，构建完整的对话上下文。

#### c. 调用大模型决策
- 通过 `generator.generateObject`，让 LLM 选择本轮 action（如直接回答、检索、反思、抓取网页、代码求解等）及其参数。

#### d. 解析 LLM 输出
- 得到本轮 action 及其参数（如 answer 内容、search 关键词、reflect 子问题等）。

#### e. 动态调整 allow* 变量
- 根据本轮 action 结果，动态调整下轮允许的 action，防止死循环和无效尝试。

---

### 4.2 各 action 分支处理

- **answer**：尝试直接回答主问题或子问题，调用评估器（evaluateAnswer）自动评估答案质量。评估通过则终止，否则生成改进建议，驱动反思或细化问题。
- **reflect**：自动生成与主问题相关的子问题，补充到 gaps，驱动多步推理。
- **search**：根据当前问题和知识，生成搜索请求，调用外部搜索引擎检索信息，补充知识库。
- **visit**：选择高相关 URL，抓取网页内容，丰富 allKnowledge。
- **coding**：遇到代码相关问题时，自动调用代码沙盒求解，补充知识库。

---

### 4.3 评估与自适应机制

- 每个问题分配多种评估类型（如完整性、时效性、权威性等）。
- 评估失败自动生成改进计划，驱动下一步反思或细化。
- 若连续多轮评估失败，自动进入 beast mode，放宽约束，优先输出任何可用答案，保证鲁棒性。

---

## 5. 答案后处理与输出

- 对最终答案进行 markdown 格式修复、引用补全、图片处理等。
- 若为聚合答案，合并所有子问题的答案。
- 返回结构化结果，包括：最终答案、推理上下文、已访问 URL、图片引用等。

---

## 6. 结果返回用户

- 通过 Web 前端、API 或命令行，将最终答案和相关信息返回给用户。
- 若为流式输出，系统会模拟“自然打字”效果，逐步将答案推送给前端。

---

# 总结

**本项目的分析流程高度自动化、模块化，核心亮点包括：**
- 问题自动分解与多步推理
- 动态决策与自适应 action 切换
- 自动评估与反思机制，保证答案质量
- 外部知识检索与内容聚合
- 强鲁棒性兜底（beast mode）

**开发者可通过 getResponse 主流程、各 action 分支、prompt 生成、评估与反思机制等关键节点，灵活扩展和优化多步推理能力。**
