基于代码分析，DeepResearch 系统实际上是一个**单一智能体系统**，而不是多智能体协作系统。 [1](#1-0) 

## 智能体架构

DeepResearch 包含一个核心研究智能体，通过不同的**工具模式**来完成复杂任务： [2](#1-1) 

### 核心智能体的五种行动模式

智能体通过动态选择以下五种行动类型来完成研究任务：

1. **search** - 搜索新信息源
2. **visit** - 访问和提取网页内容  
3. **reflect** - 生成子问题进行反思
4. **answer** - 提供最终或部分答案
5. **coding** - 执行计算任务 [3](#1-2) 

## 协作机制

虽然是单一智能体，但系统通过以下机制实现内部"协作"：

### 1. 迭代式问题分解
智能体通过 `reflect` 行动将复杂问题分解为子问题： [4](#1-3) 

### 2. 知识积累与状态管理
系统维护多个状态结构来实现"协作"：
- `gaps` - 待解决的问题队列
- `allKnowledge` - 中间知识积累
- `allURLs` - 发现的信息源
- `visitedURLs` - 已访问的资源

### 3. 专门化工具配置
不同任务使用不同的模型配置： [5](#1-4) 

- `evaluator` - 答案质量评估（温度0.6，限制200-2000 tokens）
- `coder` - 代码生成（温度0.7，使用轻量模型）
- `agent` - 核心研究（温度0.7）
- `searchGrounding` - 搜索增强（温度0）

### 4. 主循环协调
核心循环通过动态约束来协调不同行动： [6](#1-5) 

系统根据当前状态动态启用/禁用不同行动类型，实现类似多智能体的专门化行为。

## Notes

DeepResearch 采用的是"单智能体多模式"架构，而非传统的多智能体系统。这种设计通过状态管理、任务分解和专门化配置来模拟多智能体协作的效果，同时避免了多智能体间通信和协调的复杂性。

基于代码分析，DeepResearch 的核心逻辑是一个**迭代式研究循环**，通过单一智能体执行不同类型的行动来深度调研问题。

## 核心执行逻辑

### 主循环结构
系统的核心逻辑在 `getResponse()` 函数中实现，采用 while 循环持续执行直到找到答案或耗尽 token 预算： [1](#2-0) 

### 行动选择机制
每个循环步骤中，智能体通过动态 schema 生成选择最合适的行动： [2](#2-1) 

系统支持五种基础行动类型： [3](#2-2) 

## 具体行动逻辑

### 1. Search 行动
当选择搜索时，系统执行查询去重和多提供商搜索： [4](#2-3) 

### 2. Reflect 行动  
反思行动将复杂问题分解为子问题，并管理问题队列： [5](#2-4) 

### 3. Visit 行动
访问行动处理 URL 内容提取和知识积累： [6](#2-5) 

### 4. Answer 行动
答案行动包含质量评估和最终答案生成： [7](#2-6) 

### 5. Coding 行动
编程行动通过 CodeSandbox 执行计算任务： [8](#2-7) 

## 状态管理逻辑

系统维护多个关键状态来协调不同行动：
- `gaps` - 待解决问题队列
- `allKnowledge` - 中间知识积累  
- `allURLs` - 发现的信息源
- `weightedURLs` - 排序后的 URL 列表

URL 重排和过滤逻辑确保访问最相关的信息源： [9](#2-8) 

## Beast Mode 逻辑

当常规预算耗尽时，系统进入"Beast Mode"强制生成答案： [10](#2-9) 

## Notes

整个系统的逻辑核心是通过状态驱动的行动选择，结合知识积累和质量评估，实现深度研究的自动化。每个行动都有明确的触发条件和执行逻辑，通过迭代循环不断逼近最终答案。
