### 1) 定义 State（状态与阶段枚举）
- 目标：统一上下文结构，保证节点传参与条件路由可预期。
- 做法：用 TypedDict + Enum（或常量）定义状态与阶段。

```python
# 新代码示例：state/content_writing_state.py
from typing import TypedDict, List, Dict, Optional, Literal

WorkflowStage = Literal["init", "requirement", "search", "writing", "confirmation", "done"]

class ContentWritingState(TypedDict, total=False):
    current_stage: WorkflowStage
    round_count: int
    query: str
    messages: List[Dict]  # [{role, content, ...}]
    requirement_output: Dict
    search_output: Dict
    writing_output: Dict
    need_human_confirmation: bool
    human_feedback: Dict
```

要点：
- 字段尽量独立、语义明确。每个阶段产物单独挂在 `*_output`，便于审计与回放。
- 最小闭环只需 `query/messages/writing_output/current_stage` 就能跑。

### 2) 实现 Tool（原子能力）
- 目标：把“AI调用”“外部检索”“并发分析”等做成可复用纯函数/工具。
- 最小工具：仅保留“写作生成”一个工具，先跑通再扩展。

```python
# 新代码示例：tool/content_post_tool.py
from typing import Dict

def create_content_post(topic: str, style: str = "informative") -> Dict:
    # 这里先不接 LLM，返回伪结果，打通流程
    content = f"[{style}] 针对主题《{topic}》的示例文案。"
    return {"topic": topic, "style": style, "content": content}
```

要点：
- 工具函数入参出参固定为简单类型/Dict，便于被 Agent/Node 包装。
- 真实接入时，把 LLM 抽象做在 `utils/llm_client.py`，不要在工具里散落 provider 细节。

### 3) 实现 Agent（策略与工具编排）
- 目标：将 Prompt 策略 + 工具序列化使用，形成可测单元。
- 最小 Agent：一个函数，解析 query，调用 `create_content_post`。

```python
# 新代码示例：agent/content_writing_agent.py
from typing import Dict
from agents.graph.tool.content_post_tool import create_content_post

def run_content_writing_agent(state: Dict) -> Dict:
    query = state.get("query") or "通用主题"
    # 简单策略：提取写作风格，可从 state.messages 中寻找用户偏好
    style = "informative"
    result = create_content_post(topic=query, style=style)
    return {"writing_output": result}
```

要点：
- 先保证同步返回结构，后续再嵌入 ReAct/Tool Calling/多工具队列等复杂逻辑。
- 返回的 key 要与 State 对齐（`writing_output`）。

### 4) 实现 Node（图节点包装）
- 目标：把 Agent 封成图节点函数，严格输入/输出 State。
- 节点职责：读取所需状态、调用 Agent、写回状态、推进阶段。

```python
# 新代码示例：node/content_writing_node.py
from typing import Dict
from agents.graph.agent.content_writing_agent import run_content_writing_agent

def content_writing_node(state: Dict) -> Dict:
    result = run_content_writing_agent(state)
    next_state = dict(state)
    next_state.update(result)
    next_state["current_stage"] = "done"
    return next_state
```

要点：
- Node 不做复杂业务逻辑，只做“状态读写 + 调 Agent”。
- 大型系统里，Node 可负责输入校验、审计、埋点、异常兜底。

### 5) 实现 Workflow（StateGraph 编排）
- 目标：定义起点、节点、边；决定流程从 START 到 END。
- 最小工作流：`START -> content_writing -> END`

```python
# 新代码示例：workflows/content_writing_graph.py
from typing import Dict
from langgraph.graph import StateGraph, START, END
from agents.graph.node.content_writing_node import content_writing_node

def build_content_writing_graph() -> StateGraph:
    graph = StateGraph(dict)  # 简化：用 Dict 作为状态类型
    graph.add_node("content_writing", content_writing_node)
    graph.add_edge(START, "content_writing")
    graph.add_edge("content_writing", END)
    return graph

def compile_content_writing_with_memory() -> any:
    # 最小：不接 Redis，直接 compile
    graph = build_content_writing_graph()
    return graph.compile()
```

要点：
- 真实项目会用 `TypedDict` 和检查点（Memory/Redis）。先跑通 Memory 版，再接 Redis。

### 6) 编译与运行（编译图→驱动执行）
- 目标：在 API 或 CLI 封装调用，传入初始 State 执行图。
- 最小执行：

```python
# 新代码示例：demo_run_graph.py
from agents.graph.workflows.content_writing_graph import compile_content_writing_with_memory

if __name__ == "__main__":
    app = compile_content_writing_with_memory()
    init_state = {"query": "秋季护肤心得", "current_stage": "init", "messages": []}
    final_state = app.invoke(init_state)
    print(final_state["writing_output"]["content"])
```

要点：
- `invoke` 完整执行，`stream`/`astream` 可做流式节点输出，后面接 SSE。

### 7) 持久化与恢复（稍后接入）
- Memory 版：LangGraph 内存检查点，适合单机开发。
- Redis 版：结合 `redis` 检查点与 `session_id`，支持断点恢复和多轮对话。
- 真实工程：在 `workflows/*` 里提供 `compile_*_with_redis()` 变体；API 收到会话ID后选择对应编译器。

### 8) 对接 API（SSE/Schema）思路
- `/graph/chat` 接口接收 `query/session_id/user_id/...`，包装为初始 State，驱动图执行。
- SSE 推送每个节点的中间事件（推荐统一事件结构：event/type/data/timestamp）。
- 最小闭环先用 `invoke` + 一次性 JSON 输出；以后切换 `stream` 做真正 SSE。

### 9) 扩展到完整四阶段工作流
- 在 State 中加 `requirement_output/search_output`，对应实现：
  - `requirement_alignment_node`：用问答引导澄清需求，允许最多3轮
  - `content_search_node`：结合网络/向量检索（可参考你已有的增强搜索工具）
  - `content_writing_node`：收敛生成，输出草稿
  - `human_confirmation_node`：等待用户确认，并用条件路由返回 `search` 或进入 `done`
- Workflow 图：`START->supervisor->requirement->search->writing->human_confirmation->[条件路由]`
