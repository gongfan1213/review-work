### 1) 请求/事件模型要点（已具备完备度）
- Chat 请求三层结构：`ChatRequest -> RequestData`。`RequestData` 支持背景信息、交互类型、Nova3 选择、引用、文件等，可满足复杂场景。
- 流式事件三元组：`event_type + agent_source + payload`
  - `event_type` 取值在 `EventType`，如 `SYSTEM|USER|TOOL_CALL|TOOL_RETURN|LLM_CHUNK|ERROR`
  - `payload` 使用 `EventPayload`，其中 `content_type` 来自 `ContentType`，统一标注语义类型（如 `copywriting|insight|plan_step|final_answer|...`）
- 这套规范使前端不需要解析自然语言，只需按 `event_type/content_type` 路由显示。

### 3) SSE 事件发送逻辑（如何从 Graph chunk 转换）
- 路由核心流程：
  - 通过 `get_graph_app()` 编译/获取图应用（优先 Redis 混合持久化，失败回退内存）

  - 构造 `input_state`（累积 `messages`，合并 `references`）
  - 使用 `app.astream(input_state, config=..., stream_mode=["messages","updates"])` 异步迭代
  - 每个 `chunk` 经 `format_chunk_to_json` 标准化，追加到异步队列，SSE 生成器从队列消费并输出
- 队列与会话管理：
  - 异步版本 `AsyncSessionManager` 使用 `AsyncQueue` 做背压和“零轮询”，心跳每秒维持连接但不空转
  - 停止流程基于 `stop_event`，确保任务可取消且资源被清理

async def generate_async_stream():
    """异步流生成器 - 零轮询，事件驱动"""
    while True:
        try:
            response_data = await asyncio.wait_for(
                session_info.response_queue.get(), timeout=1.0
            )
            if response_data['type'] in ('stop','error','graph_end'):
                yield f"{json.dumps(response_data, ensure_ascii=False)}\n\n"
                break
            elif response_data['type'] == 'chunk':
                serializable_chunk = _make_json_serializable(response_data['data'])
                yield f"{json.dumps(serializable_chunk, ensure_ascii=False)}\n\n"
            else:
                yield f"{json.dumps(response_data, ensure_ascii=False)}\n\n"
        except asyncio.TimeoutError:
            yield f"{json.dumps({'type': 'heartbeat', 'timestamp': datetime.now().isoformat()}, ensure_ascii=False)}\n\n"

### 4) 前端/客户端消费建议
