### 二、SSE 事件模型（服务端到前端的统一契约）
建议统一事件格式，便于前端消费与排障：
- 基本字段
  - type: 阶段/节点标识，如 `init|requirement|search|writing|done|error|heartbeat`
  - data: 载荷，建议 JSON 字符串化
  - timestamp: ISO8601
  - request_id/session_id: 追踪必备（从 `core/context.require_id()` 获取）
- 建议在 `apis/schemas.py` 定义：
  - `EventType`（Literal/Enum）
  - `EventPayload`（TypedDict/Pydantic）
  - `StreamEvent`（Pydantic Model，含 type/data/timestamp/...）
- SSE 发送时保持单行 data（或多行 data: 分片），每条事件以空行分隔。

### 三、/graph/chat 的最小实现思路（从0到1）
目标：把上次讲的“最小工作流”接入到 HTTP 接口，先做“一次性 JSON 输出”，再升级为 SSE。

1) 一次性返回 JSON（最小闭环）
- 路由：`POST /graph/chat`
- 入参：`query: str, session_id: Optional[str], user_id: Optional[str], options: Optional[dict]`
- 流程：
  - 生成 `request_id`
  - 从 `settings` 读取必要参数
  - 编译最小图 `compile_content_writing_with_memory()`
  - 组装初始 State：`{"query": query, "current_stage":"init","messages":[]}`
  - `final_state = app.invoke(init_state)`
  - 返回 `final_state["writing_output"]`

2) 升级为 SSE 流式
- 使用 `sse_starlette.EventSourceResponse`
- 用 LangGraph 的 `app.stream(init_state)` 或在节点间手动生成事件
- 基本写法（伪代码要点）：
  - 在 async 生成器里 yield 心跳事件（每 10-15s）
  - 每个节点产出后 yield 一条结构化事件
  - 错误统一 try/except 包裹，yield 错误事件并结束
- 关键：确保生成器是异步的、不要阻塞（LLM/IO 用 await），并控制 backpressure（适当 `await asyncio.sleep(0)`）

3) 错误处理与超时
- 超时：可用 `asyncio.timeout` 包装节点执行或整个流程
- 错误事件：`type="error"`, `data={"message":str(e),"code":"INTERNAL"}`，同时写日志
- 终止：统一发送 `type="done"` 事件

4) 鉴权与白名单
- 若开启鉴权（Supabase/JWT），将 `/graph/chat` 保持在白名单或做基于 Header 的检查
- 限流：建议在接口级加 `get_rate_limiter()` 装饰器/依赖，防止滥用

### 四、建议的 `/graph/chat` 端到端数据流
- Request(JSON)
  - query: 用户任务
  - session_id: 会话（可空）
  - user_id: 用户（可空）
  - options: 如模型名、温度、平台等（可空）
- Backend
  - 生成 request_id
  - 记录访问计数与日志上下文
  - 编译工作流并执行（invoke/stream）
- Response
  - SSE：多条事件，最后 `type=done`
  - 或 JSON：一次性 `{writing_output: {...}}`
