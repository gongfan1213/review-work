## 🔍 **target/ 目录详细解读**

### **1. target目录的作用**
`target/`目录是Maven构建过程中生成的**输出目录**，包含编译后的代码、资源文件、测试结果等。

#### **目录结构：**
```
pmhub-monitor/target/
├── classes/           # 编译后的Java类文件
├── test-classes/      # 编译后的测试类文件
├── generated-sources/ # 自动生成的源码
├── maven-archiver/   # Maven归档信息
├── maven-status/     # Maven构建状态
├── surefire-reports/ # 测试报告
└── pmhub-monitor.jar # 最终的可执行JAR包
```

### **2. classes/ 目录详解**
这个目录包含编译后的`.class`文件，对应`src/main/java`中的Java源码。

#### **编译后的文件结构：**
```
classes/
├── com/
│   └── laigeoffer/
│       └── pmhub/
│           └── monitor/
│               ├── PmHubMonitorApplication.class
│               └── config/
│                   ├── PmHubStatusChangeNotifier.class
│                   └── WebSecurityConfigurer.class
└── application.yml
```

#### **面试要点：**
- **为什么需要编译？** Java是编译型语言，`.java`文件需要编译成`.class`字节码才能运行
- **编译过程：** `javac`编译器将源码转换为JVM可执行的字节码
- **字节码优势：** 跨平台、安全性高、性能优化

### **3. test-classes/ 目录详解**
包含测试代码编译后的`.class`文件，对应`src/test/java`中的测试源码。

#### **面试要点：**
- **测试代码分离：** 测试代码和业务代码分开编译，避免测试代码污染生产环境
- **测试覆盖率：** 可以统计测试代码对业务代码的覆盖程度
- **CI/CD集成：** 构建过程中自动运行测试，确保代码质量

## �� **pom.xml 详细解读**

### **1. 项目基本信息**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>
    <parent>
        <groupId>com.laigeoffer.pmhub-cloud</groupId>
        <artifactId>pmhub</artifactId>
        <version>0.0.1</version>
    </parent>

    <artifactId>pmhub-monitor</artifactId>
    <description>
        pmhub-monitor 监控中心，本地可访问：http://localhost:6888/wallboard，用户名：admin 密码：123456
    </description>
</project>
```

#### **逐行解读：**
- **`modelVersion`**: Maven POM模型的版本，4.0.0是当前标准版本
- **`parent`**: 继承父POM，获取公共配置和依赖管理
- **`artifactId`**: 项目标识符，用于Maven仓库中的唯一标识
- **`description`**: 项目描述，包含访问地址和默认账号信息

#### **面试要点：**
1. **为什么用parent继承？**
   - 统一版本管理，避免版本冲突
   - 继承公共配置，减少重复配置
   - 统一构建标准

2. **Maven坐标的作用？**
   - `groupId`: 组织标识，通常是公司域名反写
   - `artifactId`: 项目标识，项目名称
   - `version`: 版本号，遵循语义化版本规范

### **2. 属性配置**

```xml
<properties>
    <maven.compiler.source>8</maven.compiler.source>
    <maven.compiler.target>8</maven.compiler.target>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
</properties>
```

#### **逐行解读：**
- **`maven.compiler.source`**: 源码的Java版本，这里是Java 8
- **`maven.compiler.target`**: 编译目标的Java版本，也是Java 8
- **`project.build.sourceEncoding`**: 项目源码的字符编码，UTF-8支持中文

#### **面试要点：**
1. **为什么指定Java版本？**
   - 确保编译环境一致性
   - 避免不同JDK版本导致的兼容性问题
   - 明确项目的技术栈要求

2. **source和target的区别？**
   - `source`: 源码使用的Java语法版本
   - `target`: 生成的字节码版本
   - 通常两者保持一致

### **4. 构建配置**

```xml
<build>
    <finalName>${project.artifactId}</finalName>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <executions>
                <execution>
                    <goals>
                        <goal>repackage</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

#### **逐行解读：**
- **`finalName`**: 最终生成的JAR包名称
- **`spring-boot-maven-plugin`**: Spring Boot的Maven插件
- **`repackage`**: 重新打包，将依赖打包到JAR中

#### **面试要点：**
1. **为什么需要repackage？**
   - 将依赖的JAR包解压并重新打包
   - 生成可执行的fat JAR
   - 简化部署，一个JAR包包含所有依赖

2. **Maven插件的作用？**
   - 扩展Maven的构建能力
   - 提供特定的构建任务
   - 集成第三方工具
